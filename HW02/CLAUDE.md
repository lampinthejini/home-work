# 프로젝트 안내

## 개요
이 프로젝트는 연락처 관리 웹 앱이다.

## 코딩 규칙
- 들여쓰기는 공백 4칸을 쓴다.
- 함수와 변수 명은 영어 카멜 표기법을 사용한다.
- SDD,TDD 방식으로 개발한다.

## 검증 명령
-- 테스트 : python -m pytest
-- 형식 검사 : lint 사용

## 안드레이 카르파시(Andrej Karpathy) 행동 원칙
1. 코딩하기 전에 먼저 생각하라 (Think Before Coding)

추측하지 말라. 헷갈리는 부분을 숨기지 말라. 여러 선택지가 있다면 그 차이를 드러내라.

구현을 시작하기 전에 다음을 수행하라.

자신의 가정을 명확하게 밝힌다.
확신이 없다면 질문한다.
여러 해석이 가능하다면, 조용히 하나를 선택하지 말고 가능한 해석들을 제시한다.
더 단순한 접근법이 있다면 그것을 알려준다.
필요하다면 현재 방식에 이의를 제기한다.
무언가 명확하지 않다면 구현을 멈춘다.
무엇이 애매한지 설명하고 질문한다.

2. 단순함을 최우선으로 하라 (Simplicity First)

문제를 해결하는 데 필요한 최소한의 코드만 작성하라.

추측에 기반한 기능은 만들지 말라.

요청받지 않은 기능은 추가하지 않는다.
한 번만 사용할 코드를 위해 추상화를 만들지 않는다.
요청되지 않은 확장성이나 설정 기능을 넣지 않는다.
현실적으로 발생하지 않는 상황을 위해 과도한 예외 처리를 하지 않는다.
200줄을 작성했는데 50줄로 해결할 수 있다면 다시 작성한다.

항상 스스로에게 질문하라.

"이 코드를 본 시니어 엔지니어가 '너무 복잡하다'고 말하지 않을까?"

그렇다면 더 단순하게 만든다.

3. 필요한 부분만 정확하게 수정하라 (Surgical Changes)

정말 필요한 부분만 수정하라.

그리고 자신이 만든 흔적만 정리하라.

기존 코드를 수정할 때는 다음 원칙을 따른다.

관련 없는 코드나 주석, 포맷을 함부로 개선하지 않는다.
문제가 없는 코드를 리팩터링하지 않는다.
자신의 취향이 아니라 기존 프로젝트의 스타일을 따른다.
관련 없는 죽은 코드(dead code)를 발견했다면 알려주기만 하고 삭제하지 않는다.

자신의 변경으로 인해 사용되지 않게 된 것들은 정리한다.

자신이 만든 미사용 import, 변수, 함수는 제거한다.
기존부터 존재하던 미사용 코드는 요청받지 않는 이상 건드리지 않는다.

항상 다음 기준을 만족해야 한다.

"수정된 모든 코드 한 줄 한 줄은 사용자의 요청과 직접 연결되어 있어야 한다."

4. 목표 중심으로 실행하라 (Goal-Driven Execution)

성공 기준을 먼저 정의하고, 검증될 때까지 반복하라.

모든 작업을 검증 가능한 목표로 바꾼다.

예를 들어

"유효성 검사를 추가하라."
→ "잘못된 입력에 대한 테스트를 먼저 작성하고, 그 테스트가 통과하도록 구현한다."
"버그를 수정하라."
→ "버그를 재현하는 테스트를 먼저 작성한 뒤, 그 테스트가 통과하도록 수정한다."
"X를 리팩터링하라."
→ "리팩터링 전후 모두 테스트가 통과하는지 확인한다."

여러 단계의 작업이라면 간단한 계획을 먼저 제시한다.

작업 수행 → 검증 방법
작업 수행 → 검증 방법
작업 수행 → 검증 방법

강력한 성공 기준이 있으면 스스로 반복하며 작업을 진행할 수 있다.

반대로

"그냥 동작하게 만들어라."

같은 모호한 목표는 계속해서 추가 확인과 질문을 필요로 하게 된다.
